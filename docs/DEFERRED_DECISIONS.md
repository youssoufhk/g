# Deferred Decisions Registry

> **What this is:** a living registry of decisions where Gamma consciously chose the simpler or cheaper path in v1.0 for a specific reason, with an explicit trigger to reopen and a rough cost estimate for future implementation.
>
> **What this is not:** a roadmap or Phase 6+ feature list. Roadmap items are planned work. The items here are **tradeoffs we are actively accepting today** and need to remember the context behind.

## How to use this registry

- **Status column:** the leftmost column has two values: "Open" (active deferral) or "Resolved (YYYY-MM, PR #NNN)". Open rows are active tradeoffs; Resolved rows are kept for history. When scanning the registry manually, sort Open rows above Resolved.
- **Stable IDs:** every item has a `DEF-NNN` ID that does not change. Reference the ID from code comments, commit messages, PR descriptions, ADRs, and future planning sessions. Example in code:
  ```python
  # DEF-021: Migration off Workspace SMTP Relay to a dedicated provider
  # when quota pressure appears. Until then we send through smtp-relay.gmail.com.
  def send_transactional_email(...):
      ...
  ```
- **Triggers:** each item documents the concrete event that should cause the team to reopen the decision. Ignore the item until its trigger fires.
- **Cost estimates:** rough work-weeks, not commitments. Re-estimate at the moment of reopening because dependencies shift.
- **Adding new items:** when a future decision deliberately defers work, append it to the table with the next `DEF-NNN` ID, same five columns. Do not delete or reorder existing rows even when they are resolved; mark them "RESOLVED in YYYY-MM" in the "Chosen path" column.
- **Resolving an item:** when the trigger fires and the team implements the deferred work, prefix the row with `[RESOLVED: YYYY-MM, PR #NNN]` and keep it. The registry is append-only for historical tracing.

## How this registry was created

The initial 59 items (DEF-001 through DEF-059) were extracted from a multi-round data architecture planning session that locked 102 decisions before Phase 2. After extraction, the planning session's working document was deleted and this registry became the permanent reference. Later items (DEF-060 onward) were added as data ingestion and other specs were drafted.

Every `DEF-NNN` below was a conscious founder-approved tradeoff, not a forgotten item. The "chosen path in v1.0" column describes what Gamma actually does in v1.0 so that future-self can judge whether the original reason still holds without needing any other document.

## Summary by category

**AI / model / vendor (DEF-038 to DEF-046):** nine items all orbiting the Vertex AI Gemini choice. Evals in CI are the safety net; the single `ai/client.py` abstraction is the reversibility escape hatch.

**Billing and pricing (DEF-028 to DEF-032, DEF-059):** six items. Manual invoicing in Phase 2 is the Phase-5 upgrade path; Revolut vs Stripe vs Paddle evaluation deferred to trigger time.

**GDPR and legal (DEF-033 to DEF-037):** five items. The core GDPR posture (residency, DPA, DSR self-service, retention) is in v1.0; these are the enterprise-grade extras.

**Auth and identity (DEF-008, DEF-024, DEF-025):** three items. OIDC + passkey ship Phase 2; SSO-related deferrals (portal SSO, SCIM, SAML) all trigger on specific customer demands.

**Operator console (DEF-002 to DEF-004, DEF-054, DEF-056):** five items. v1.0 operator console is tenant-lifecycle-only; richer features wait for real ops pain.

**Infrastructure and ops (DEF-013 to DEF-023, DEF-048 to DEF-053, DEF-055 to DEF-058):** ~20 items. All are cost/complexity deferrals where the v1.0 path is "boring, cheap, well-known" and the upgrade path is "when scale or pain demands it."

**Feature scope (DEF-001, DEF-005 to DEF-012, DEF-047, DEF-050, DEF-051):** ~13 items. v1.0 ships the 80% product; these are the 20% that a specific customer will eventually ask for.

**Data ingestion (DEF-060 to DEF-064):** five items added after the DATA_INGESTION spec was written. External HRIS sync, historical non-timesheet imports, bidirectional payroll sync, Excel format support, and Google Sheets direct import. All deferred because CSV BYO covers the first-customer case.

**Country expansion (DEF-071 to DEF-074):** four items covering year 2 markets beyond the FR + UK baseline. Canada (PIPEDA + Quebec Law 25), Morocco (CNDP + Arabic RTL + CMI), Niger (CNPDCP + WAEMU + mobile money), and the WAEMU shared pattern after Niger. Per-country strategy modules and the `tenants.residency_region` / `legal_jurisdiction` columns are in place in v1.0 so expansion is a config + one-file change per country, not a refactor. See `docs/COUNTRY_PLAYBOOKS.md` for reference sheets.
